Documentação técnica oficial

BRazucaFlow API

Referência técnica para integração, testes e publicação dos endpoints da API financeira do BRazucaFlow.

Versão1.0 - base atual do projeto
FormatoREST / JSON
AtualizaçãoJunho/2026

Visão Geral

A API BRazucaFlow centraliza as operações financeiras do sistema: usuários, perfis, plano de contas, contas a pagar, contas a receber, receitas, despesas, auditoria, backups e dashboard.

Esta documentação descreve os endpoints disponíveis, parâmetros aceitos, exemplos de payload e regras relevantes para consumo da API.

Base URL
Em desenvolvimento local, a API normalmente responde em http://localhost:3000. Em produção, substitua pela URL pública configurada para o servidor.

Autenticação

A API utiliza autenticação por token JWT. Após autenticar o usuário pela rota de login, envie o token nas demais requisições usando o cabeçalho HTTP abaixo:

Authorization: Bearer <token_jwt>

A rota de login é pública. Todas as demais rotas exigem usuário autenticado, e algumas são restritas ao perfil Super Admin.

Padrão de Resposta

Sucesso

{
  "success": true,
  "data": {},
  "status": 200
}

Erro

{
  "success": false,
  "error": "Descrição do erro",
  "status": 400
}
StatusUso
200Consulta, atualização, exclusão ou ação executada com sucesso.
201Registro criado com sucesso.
400Validação ou regra de negócio impediu a operação.
401Token ausente, token inválido ou credenciais inválidas.
403Usuário autenticado sem permissão para a operação.
404Registro não encontrado.
409Conflito de dados, como registro duplicado.
500Erro interno inesperado.

Convenções

ItemDescrição
PaginaçãoQuando disponível, use page e limit. O limite máximo aplicado pela API é 100 registros por página.
OrdenaçãoUse sortBy e sortDirection, conforme os campos aceitos em cada módulo.
DatasEnvie datas no formato YYYY-MM-DD.
Valores monetáriosEnvie números em formato decimal, por exemplo 1500.75.
BooleanosA API aceita booleanos em campos como paid, status, person e similares.
IDs de exemploOs IDs nos payloads são exemplos. Devem ser substituídos por registros existentes no banco.
Segurança: esta documentação não deve conter usuários reais, senhas, tokens ou dados sensíveis. Para testes, configure credenciais em ambiente seguro.
